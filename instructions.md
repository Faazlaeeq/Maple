# Development Brief: "Maple" AI Front Desk Assistant
**Document Owner:** Engineering Lead
**Project Codename:** Maple
**Status:** Ready for Development
**Audience:** Full-stack engineer(s) building this from scratch

---

## 1. Purpose of This Document

This is the complete specification for building Maple, an embeddable AI chat assistant for small local service businesses (initial target: dental practices). Everything a developer needs to build this without further clarification should be in this document. If something isn't covered here, that's a gap to flag before writing code, not to assume around.

---

## 2. Business Context (Read This Before Anything Else)

The end client is a non-technical small business owner (e.g., a dentist) who already has a website built and hosted by a third party they have no direct access to or understanding of. **This single constraint drives several requirements below** — we cannot assume any ability to modify the host site beyond pasting one snippet of code. Design accordingly: the embed mechanism is not a nice-to-have, it is the delivery mechanism for the entire product.

The core value proposition: capture and respond to website visitor inquiries 24/7, log every interaction, and notify the business owner immediately of anything requiring follow-up — without the owner needing to understand or touch any of the underlying technology.

---

## 3. Scope

### In Scope (v1 / MVP)
- Embeddable chat widget, deployable via a single `<script>` tag
- AI-powered conversation grounded in a per-client knowledge base (FAQs, services, hours, policies)
- Lead capture (name, contact info, inquiry summary) when the assistant can't fully resolve a query or the visitor wants to book
- Persistent conversation logging
- Real-time email notification to the business owner on every captured lead
- Basic admin visibility into captured leads (can be a simple authenticated page, does not need to be polished in v1)

### Explicitly Out of Scope (v1)
- Native calendar booking (Google Calendar/Calendly API integration) — design the data model to support this later, but do not build it now
- Multi-language support
- Voice/audio input
- Multi-tenant admin dashboard with billing, user management, etc. — v1 is single-client, hardcoded configuration is acceptable
- Any CMS or self-serve way for the client to edit their own knowledge base — that's a v2 problem; v1 knowledge base is a config file you edit directly

Do not build ahead of this scope. A working, reliable v1 within scope is the deliverable — not a more ambitious partial system.

## 4. Functional Requirements

### 4.1 Widget Behavior
- Chat bubble fixed to bottom-right corner of the viewport on every page of the host site
- On first page load per session, after a 3-4 second delay, display a proactive greeting bubble (not a full open chat window) with a short, specific message — this is a nudge, not an interruption
- Clicking the bubble expands a chat window (mobile-responsive — must degrade gracefully to full-width on small viewports, not just shrink)
- Conversation state persists within a browser session (if the visitor navigates between pages on the host site, the conversation should not reset) — use sessionStorage or a session token, not a full backend session requirement
- Widget must visually indicate when the assistant is "typing" (a response is pending) — no silent multi-second waits with no feedback

### 4.2 Conversation / AI Behavior
- Every response is generated by calling the Gemini API (model: `gemini-2.5-flash-lite` for v1 — see Section 6 for rationale) with a system prompt plus the client's knowledge base plus the running conversation history
- The assistant must never fabricate information not present in the knowledge base. If a question falls outside the knowledge base, the correct behavior is to say so honestly and pivot to capturing contact info for human follow-up — this is a hard requirement, not a style preference
- The assistant must never provide clinical/medical advice of any kind, regardless of how the question is phrased. Any question that reads as seeking a diagnosis or treatment recommendation must be redirected to booking an appointment
- Response length: 2-3 sentences maximum per turn. If a genuinely longer answer is needed, break it into a follow-up rather than one large block
- The assistant should ask at most one clarifying question per turn, never stack multiple questions in one message
- The assistant must proactively attempt to capture contact information (name + phone or email) whenever: (a) it cannot fully resolve the inquiry, or (b) the visitor expresses intent to book/visit

### 4.3 Lead Capture & Data Handling
- The moment a visitor provides a name and any contact method (email or phone), a lead record must be created — do not wait for the conversation to end
- Every lead record must store: full conversation transcript up to that point, visitor-provided contact info, a timestamp, and an auto-generated one-line summary of the inquiry (this can be generated via a second lightweight Gemini call, or extracted via prompt instruction in the same call — engineer's choice, but the summary field must be non-empty)
- Leads must be flagged with an urgency indicator (`normal` / `urgent`) — determine this via prompt instruction based on conversation content (e.g., "emergency," "pain," "urgent," "today" should trigger `urgent`)

### 4.4 Notifications
- On every new lead record creation, send an email notification to a configured business owner address within seconds — this must not be a batched/delayed job, it needs to feel real-time
- Email content must include: visitor name, contact info, one-line summary, urgency flag, and a link to view the full transcript
- Notification failures (e.g., email service downtime) must be logged server-side and must not crash or block the conversation flow for the visitor — the visitor-facing experience must never depend on notification delivery succeeding

### 4.5 Embed Mechanism (Critical — Read Carefully)
- The entire widget must be deliverable via a single `<script src="...">` tag that the client (or their existing web developer/agency) can paste anywhere in their site's HTML, ideally before `</body>`
- The script must inject its own container element and must **not** depend on any existing element, class, or ID on the host page — assume zero cooperation or awareness from the host site's codebase
- **CSS isolation is mandatory.** The widget's styles must not leak into or be affected by the host site's styles. Use a Shadow DOM for the widget's root element, or heavily namespaced/scoped CSS with `!important` overrides as a fallback if Shadow DOM introduces complications with the chosen chat library
- The script bundle must be small and must load asynchronously — it must never block or measurably slow down the host site's page load. Target under 150KB gzipped for the full widget bundle
- The script must fail silently and invisibly if it cannot load or initialize (e.g., network issue) — it must never throw visible errors on the host page or break other functionality on that page

## 5. Technical Architecture

```
┌─────────────────────┐
│   Host website       │  (third-party built, no access)
│   <script> tag only   │
└──────────┬───────────┘
           │ loads
           ▼
┌─────────────────────┐
│  Widget bundle        │  React, built + hosted by us
│  (Shadow DOM root)    │  Served from CDN/Vercel
└──────────┬───────────┘
           │ REST calls
           ▼
┌─────────────────────┐
│  Backend API          │  Node.js / Express
│  (our infrastructure) │  Hosted on Vercel or Railway
└──────────┬───────────┘
           │
     ┌─────┴──────┬─────────────┐
     ▼             ▼             ▼
┌─────────┐  ┌───────────┐  ┌──────────────┐
│ Gemini  │  │ Firestore │  │ Email service │
│ API     │  │ (leads +  │  │ (Resend or    │
│         │  │ transcripts)│  Nodemailer)  │
└─────────┘  └───────────┘  └──────────────┘
```

### 5.1 Frontend Widget
- React, bundled as a standalone script (Vite library mode or similar — do not ship the whole app framework, ship a minimal bundle)
- Shadow DOM encapsulation as described in 4.5
- Communicates with backend via a small number of REST endpoints (see 5.3)

### 5.2 Backend API
- Node.js/Express
- Stateless request handling — conversation history is passed from the client on each request (do not require sticky sessions or in-memory state that breaks on redeploy/scaling)
- Responsible for: constructing the full prompt (system prompt + knowledge base + history), calling Gemini, parsing/validating the response, writing to Firestore, triggering notifications

### 5.3 API Contract (minimum required endpoints)
- `POST /api/chat` — accepts `{ sessionId, message, history[] }`, returns `{ reply, leadCaptured: boolean }`
- `POST /api/lead` — (may be internal/called server-side only, not necessarily a public endpoint) writes a lead record and triggers notification
- `GET /api/admin/leads` — authenticated, returns list of captured leads for the simple admin view (basic auth or a shared secret token is acceptable for v1 — full user auth system is out of scope)

### 5.4 Data Model (Firestore)

**Collection: `leads`**
```
{
  id: string,
  clientId: string,          // hardcoded per-deployment in v1
  visitorName: string,
  contactMethod: "email" | "phone",
  contactValue: string,
  summary: string,
  urgency: "normal" | "urgent",
  transcript: [{ role: "user" | "assistant", text: string, timestamp: Timestamp }],
  createdAt: Timestamp,
  notificationSent: boolean
}
```

**Collection: `conversations` (optional — may be folded into `leads` if simpler, engineer's call)**
Full conversation logs even for sessions that never convert to a lead, for later analysis of what visitors are asking that the knowledge base doesn't cover.

### 5.5 Knowledge Base
- v1: a single structured JSON or YAML file per client, containing services, FAQs, hours, policies, pricing ranges
- This file is loaded server-side and injected into the system prompt on every request — no vector database, no embeddings, no RAG pipeline needed at this scale. Do not over-engineer this.

## 6. AI Integration Requirements

- **Model:** `gemini-2.5-flash-lite`. Rationale: at realistic small-business conversation volume (~20/day), monthly API cost is under $0.20 — cost is a non-issue at this model tier, and it is sufficient for scoped FAQ + lead-capture behavior. Do not default to a larger/more expensive model without a specific, demonstrated quality failure that justifies it.
- The system prompt must be treated as a first-class engineering artifact, not an afterthought — version it, keep it in a dedicated file (not inline in application code), and write it to enforce every behavior in Section 4.2 explicitly (not just imply it)
- The knowledge base must be injected in full into every request (not chunked/retrieved) given its small size — simplicity over premature optimization
- All Gemini API calls must have a timeout (recommend 8-10 seconds) and a graceful fallback message if the call fails or times out (e.g., "I'm having trouble responding right now — can you leave your contact info and we'll follow up?") — this fallback is itself a lead-capture opportunity, not just an error state

---

## 7. Security & Privacy Requirements

- All visitor-submitted contact information must be transmitted over HTTPS only — non-negotiable
- The admin leads endpoint must be authenticated — no publicly accessible lead data under any circumstance
- No visitor data should be logged to third-party analytics or logging services beyond what's necessary for debugging (application logs are fine; do not pipe transcripts to any external analytics tool)
- API keys (Gemini, email service) must be stored as environment variables, never committed to source control, never exposed to the frontend bundle

---

## 8. Performance Requirements

- Widget script load must not add more than ~200ms to host page load time, and must load asynchronously so it never blocks host page rendering
- Chat response latency target: under 3 seconds from message send to reply displayed, under normal Gemini API conditions
- Widget must function correctly on the last two major versions of Chrome, Safari, and Firefox, and on mobile Safari/Chrome

---

## 9. Testing & Acceptance Criteria

Before this is considered done, the following must all be verifiably true:

1. The widget can be embedded on a completely unrelated, unmodified static HTML page using only the provided script tag, with zero visual or functional interference with that page's existing content
2. A full conversation (greeting → question → answer → lead capture) can be completed end-to-end, and the resulting lead record appears correctly in Firestore within seconds
3. An email notification arrives within seconds of lead capture, containing all required fields from Section 4.4
4. Asking a question outside the knowledge base produces an honest "I don't know, let me get your info" response — not a fabricated answer
5. Asking a clinical/medical question produces a redirect to booking, never a diagnostic-sounding response
6. The widget degrades correctly on a mobile viewport (375px width minimum)
7. Killing the Gemini API call (simulate a timeout) produces the graceful fallback message, not a broken UI state or unhandled error
8. The admin leads view requires authentication and correctly displays all captured leads with transcripts

---

## 10. Deployment Requirements

- Widget bundle and backend API both deployable to Vercel (or Railway if Vercel's function timeout limits become an issue with Gemini call latency — engineer's judgment call, document the choice)
- Environment-based configuration (dev/prod) for API keys and the notification recipient email
- A README documenting: how to update the knowledge base file, how to redeploy after changes, and the exact embed snippet to hand to a client

## 12. Definition of Done

This project is done when a non-technical person could hand the embed script to any web developer, have it pasted into an unrelated site in under 5 minutes, and from that point forward every visitor conversation is handled correctly, every lead is captured and logged, and the business owner receives a real-time email — without you touching anything further. If any part of that chain requires manual intervention, it is not done yet.
